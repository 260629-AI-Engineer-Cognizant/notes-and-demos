# Chroma: Local Vector Search

## Big Idea

Chroma is a local/open-source vector database that is useful for learning, prototyping, and building small semantic search demos. It lets us create collections, add documents and embeddings, query by similarity, and filter using metadata.

---

## Why Use Chroma?

Chroma is useful because it can run locally or inside a notebook.

Good for:

- learning vector database concepts
- local semantic search demos
- small RAG prototypes
- testing chunking and metadata strategies
- avoiding paid cloud setup during training

Less ideal for:

- large-scale production workloads
- organization-wide managed infrastructure
- strict cloud governance needs

---

## Chroma Terminology

| Concept | Meaning |
|---|---|
| Client | Connection to Chroma |
| Collection | Group of related vector records |
| Document | Original text being stored |
| Embedding | Numeric vector representing the document |
| Metadata | Extra filterable information about the document |
| ID | Unique identifier for each record |

---

## In-Memory vs Persistent Chroma

### In-Memory Client

Stores data temporarily.

Good for quick demos.

Once the notebook restarts, the data is gone.

### Persistent Client

Stores data to disk.

Good when we want the database to remain after the script or notebook ends.

Example idea:

```python
client = chromadb.PersistentClient(path="./chroma_week3_db")
```

---

## Basic Workflow

```text
1. Create/load documents.
2. Generate embeddings.
3. Create a Chroma collection.
4. Add or upsert records.
5. Query with a user question.
6. Filter using metadata when needed.
7. Update or delete records when data changes.
```

---

## Creating a Collection

A collection stores related vector records.

Example:

```python
collection = client.get_or_create_collection(name="week3_ai_notes")
```

Think of it like a table for vector search.

---

##Adding Records

A Chroma record can include:

```python
collection.add(
    ids=["doc-001"],
    documents=["CNNs are useful for image classification."],
    embeddings=[[0.12, -0.33, 0.87]],
    metadatas=[{"topic": "tensorflow", "week": 2}]
)
```

In real projects, the embedding vector is generated by an embedding model.

---

## Querying Records

To query semantically:

```python
collection.query(
    query_embeddings=[query_embedding],
    n_results=3
)
```

The query text must be embedded using the same embedding model used for the stored documents.

Important: vector dimensions must match. If the collection stores 384-dimensional embeddings, the query embedding must also have 384 dimensions.

---

## Metadata Filtering

Metadata filtering narrows the search.

Example:

```python
collection.query(
    query_embeddings=[query_embedding],
    n_results=3,
    where={"topic": "tensorflow"}
)
```

This means:

```text
Find semantically similar results, but only inside TensorFlow documents.
```

---

## Updating and Deleting

Data changes over time. Vector databases need basic data management.

Common operations:

| Operation | Meaning |
|---|---|
| Add | Insert new records |
| Upsert | Insert or replace records |
| Update | Change existing record values |
| Delete | Remove records |
| Get | Retrieve records by ID or filter |
| Query | Similarity search |

## Chroma for Recommendation Systems

Chroma is not only useful for semantic search. The same vector similarity workflow can also power simple recommendation systems.

Semantic search usually starts with a user query:

```text
User query → query embedding → similar document/product vectors
```

A recommendation system can use a different type of query vector:

```text
Viewed product → product embedding → similar product embeddings
```

or:

```text
User's liked items → averaged user-profile embedding → recommended item embeddings
```

This means a vector database can support more than one retrieval pattern.

| Pattern | Query Vector Comes From | Example |
|---|---|---|
| Semantic search | User's typed question/search | "gear for trail running" |
| Similar item recommendation | A product/document the user viewed | "more like this hiking shoe" |
| Personalized recommendation | A user profile built from liked/viewed items | "recommend items for this user's interests" |

---

### Similar-Item Recommendation

In an item-to-item recommender, the selected item becomes the query.

Example:

```text
User viewed:
Trail Runner Pro Shoes

Find:
Products with embeddings close to Trail Runner Pro Shoes
```

Code idea:

```python
viewed_product = collection.get(ids=["prod-001"], include=["embeddings"])

results = collection.query(
    query_embeddings=[viewed_product["embeddings"][0]],
    n_results=5
)
```

This is still vector similarity search, but instead of embedding a new search phrase, we reuse the vector for an existing item.

---

### User-Profile Recommendation

A basic personalized recommender can average the embeddings of items a user liked, viewed, or purchased.

Example:

```text
User liked:
- Trail Runner Pro Shoes
- Hydration Running Backpack
- GPS Running Watch

Average those vectors to create a user profile vector.

Then search for nearby product vectors.
```

Code idea:

```python
import numpy as np

liked_items = collection.get(
    ids=["prod-001", "prod-004", "prod-009"],
    include=["embeddings"]
)

user_vector = np.mean(liked_items["embeddings"], axis=0)

results = collection.query(
    query_embeddings=[user_vector.tolist()],
    n_results=5
)
```

The result is a simple recommendation list based on similarity to the user's interests.

---

### Why Include Category and Tags in Embedded Text?

When building product embeddings, we often combine several fields into one text string.

Example:

```text
Product: Trail Runner Pro Shoes
Category: Footwear
Tags: trail running, hiking, grip, outdoors
Description: Lightweight shoes with grippy soles for uneven terrain.
```

We do this because the embedding model only understands the text it receives.

If category and tags should influence similarity, include them in the embedded text.

If category and tags should only be used for filtering, store them as metadata.

Common pattern:

```text
category/tags in embedded text → influence semantic similarity
category/tags in metadata → support filtering
```

Both are useful.

---

### Metadata Filtering in Recommendations

Metadata can make recommendations more controlled.

Example:

```python
collection.query(
    query_embeddings=[user_vector.tolist()],
    n_results=5,
    where={"category": "footwear"}
)
```

This means:

```text
Recommend items similar to the user profile,
but only return footwear.
```

Metadata filters are helpful when we want recommendations within a specific category, price range, difficulty level, source, or business rule.

---

### Recommendation System Takeaway

A vector database can support recommendation systems because recommendations are often a similarity problem.

Instead of asking:

```text
Which document is most similar to this question?
```

we ask:

```text
Which products are most similar to this product?
```

or:

```text
Which products are most similar to this user's interests?
```

This is the same core idea as semantic search:

```text
Create embeddings → store vectors → query by similarity → return nearest results
```

## Common Mistakes

### Mistake 1: Using Different Embedding Models

If documents use one embedding model and queries use another, results may be poor or fail due to dimension mismatch.

---

### Mistake 2: Forgetting Metadata

Without metadata, every query searches everything. Metadata helps control retrieval.

---

### Mistake 3: Chunking Too Large or Too Small

Huge chunks may contain too many unrelated ideas.

Tiny chunks may lose context.

---

### Mistake 4: Expecting Semantic Search to Be Perfect

Semantic search finds likely matches, not guaranteed answers. Results should be inspected and evaluated.

---

## Bite-Sized Version

Chroma is a local vector database. We use it to store embeddings, documents, IDs, and metadata. A Chroma collection is like a vector-search table. We can add or upsert records, query by semantic similarity, filter with metadata, and update/delete records. The same similarity workflow can also support recommendation systems by querying with a product vector or a user-profile vector.
